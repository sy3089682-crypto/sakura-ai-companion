# Contributing to Sakura AI Companion

Thank you for your interest in contributing! This document provides guidelines and instructions for contributing to the project.

## Code of Conduct

- Be respectful and inclusive
- Provide constructive feedback
- Respect the project's vision and scope
- Help others learn and grow

## How to Contribute

### Reporting Bugs

**Before submitting a bug report:**
- Check if the issue already exists
- Test with the latest code
- Gather as much information as possible

**Include in your bug report:**
- Clear title and description
- Steps to reproduce
- Expected vs actual behavior
- Browser/environment details
- Screenshots if applicable
- API key issues (without sharing your actual key)

### Suggesting Enhancements

**Before suggesting an enhancement:**
- Check if it's already been suggested
- Verify it aligns with project scope

**Include in your suggestion:**
- Clear title and description
- Why this enhancement would be useful
- Possible implementation approach
- Examples of similar features in other projects

### Pull Requests

1. **Fork the repository**
   ```bash
   git clone https://github.com/yourusername/sakura-ai-companion.git
   cd sakura-ai-companion
   ```

2. **Create a feature branch**
   ```bash
   git checkout -b feature/your-feature-name
   ```

3. **Make your changes**
   - Write clear, commented code
   - Follow existing code style
   - Test thoroughly in multiple browsers
   - Keep commits atomic and descriptive

4. **Commit with clear messages**
   ```bash
   git commit -m "Add: descriptive message of changes"
   ```

5. **Push to your fork**
   ```bash
   git push origin feature/your-feature-name
   ```

6. **Open a Pull Request**
   - Use the provided PR template
   - Reference related issues
   - Describe your changes clearly
   - Include testing details

## Coding Standards

### JavaScript
- Use ES6+ features
- Write descriptive variable/function names
- Add comments for complex logic
- Keep functions focused and small
- Avoid global variables when possible

### CSS
- Use meaningful class names
- Organize styles logically
- Include comments for complex selectors
- Maintain responsive design principles
- Test across browsers

### HTML
- Use semantic HTML5 elements
- Maintain accessibility standards
- Keep markup clean and organized

## Testing

- Test in multiple browsers (Chrome, Firefox, Safari, Edge)
- Test on mobile and desktop
- Verify localStorage functionality
- Test API integration with test keys
- Check dark mode switching
- Verify typing effects and animations

## Documentation

- Update README.md for user-facing changes
- Document new features and settings
- Keep comments current
- Include examples when helpful

## Review Process

- At least one maintainer review required
- CI checks must pass
- Code should follow project standards
- Changes should align with project vision

## Development Tips

- Use browser DevTools for debugging
- Check console for errors
- Test with slow internet connections
- Verify localStorage limits
- Consider accessibility (WCAG guidelines)

## Questions?

- Check existing issues and discussions
- Ask in a new issue or discussion
- Be patient and respectful
